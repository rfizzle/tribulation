Most releases ship AI-written notes: the release workflow rewrites the merged
pull requests into player-facing prose. A **hand-authored changelog overrides
that** ‚Äî commit `changelogs/<version>.md` (the version with no `v` prefix, e.g.
`changelogs/1.0.0.md`) and the workflow publishes it **verbatim** to GitHub,
Modrinth, and CurseForge and does not run the model. Reach for it when a release
deserves curation the PR list can't give it: a `1.0.0` milestone, a headline
feature, a breaking change players must act on.

Because it ships verbatim to three store fronts, the file *is* the public release
notes. There is no editor between you and the player ‚Äî write it as the finished
artifact.

## Who reads it, and from where

Players, on a store page or the in-launcher update card ‚Äî not developers reading
a diff. They are deciding "should I update, and what changes when I do." The
changelog answers that and nothing else. It renders as Markdown on all three
platforms (CurseForge is told `changelogType: markdown`).

## Shape

Open with **one or two sentences** naming what the release is about, then group
changes under the three fixed headings. Omit any section that would be empty.

```markdown
Trading stalls get a full rework this release, plus the crash some of you hit on
empty stalls is gone.

## Added
- Stall search bar ‚Äî filter a villager's offers by item name.
- A "favorite" star on any offer; favorites float to the top.

## Changed
- Restock timer shortened from 20 to 12 minutes.
- Trade GUI now shows the exact emerald count, not a rounded stack icon.

## Fixed
- Crash when opening a stall with no stocked offers.
- Prices no longer reset after a server restart.
```

Headings are exactly `## Added`, `## Changed`, `## Fixed`, in that order ‚Äî the
same vocabulary the AI notes use, so a reader can't tell a hand-written release
from a generated one by its shape. Keep it tight: a handful of bullets per
section at most. If a section would run long, you are probably listing commits,
not changes ‚Äî collapse them to the player-visible outcome.

## What goes in

- **Player-visible changes only** ‚Äî new content, changed behavior, fixed bugs
  someone actually hit. If a player would never notice it, it does not belong.
- **The outcome, in their words.** "Villagers restock twice as fast," not
  "reduced `RESTOCK_INTERVAL` constant." Name the block/item/screen as it appears
  in-game.
- **Action they must take**, up front, when a release needs it ‚Äî a config reset,
  a world backup, a now-incompatible dependency. Breaking changes lead; players
  should not discover them from a crash.
- **The lead sentence** ‚Äî the one thing to know before the bullets. It is the
  part most likely to be read.

## What stays out

- **Developer/CI jargon:** commit hashes, PR numbers, branch names, author
  handles, "bumped Loom," "refactored the registry," version bumps of build
  tooling, green-CI notes. None of it means anything on a store page.
- **Invented or inflated changes.** Describe only what actually shipped in this
  version ‚Äî never round a small fix up into a headline feature, and never list
  something you *plan* to do. The changelog is a record, not a roadmap.
- **Internal-only work** with no player-facing effect: test coverage, code
  cleanup, doc edits, mixin reshuffles. Real work, wrong audience ‚Äî it lives in
  the git history, not the release notes.
- **Duplication of the store description.** The changelog is what changed *this
  version*, not a re-pitch of the whole mod (that is `site/listing-*.md`).
- **Sign-offs, preambles, and tool banners** ‚Äî no "Here are the release notes,"
  no "Generated by," no "Thanks for playing" footer. Start at the lead sentence,
  end at the last bullet.

## Version-scheme fit

The filename must match the release tag exactly, minus the `v`: tag `v1.2.0` ‚Üí
`changelogs/1.2.0.md`. A pre-release keeps its suffix ‚Äî tag `v1.0.0-rc1` ‚Üí
`changelogs/1.0.0-rc1.md`. A mismatched or missing filename means the workflow
never sees the file and silently falls back to AI notes, so the release ships
generated prose instead ‚Äî check the name before tagging. See the **Version
scheme** and **Release notes** sections of `AGENTS-COMMON.md`.

## Guardrails

- **Always** write for players: describe the in-game outcome, never the code
  change or the commit that made it.
- **Always** name the file `changelogs/<version>.md` with the exact tag version
  minus `v`, or the workflow falls back to AI notes without warning.
- **Never** include commit hashes, PR numbers, author handles, branch names, or
  CI/build jargon ‚Äî they render verbatim on the store page.
- **Never** invent, inflate, or pre-announce changes; list only what shipped in
  this version.
- **Never** add a preamble, sign-off, or "generated by" banner ‚Äî the file ships
  exactly as written, so its first line is the first line players read.
- Use only `## Added`, `## Changed`, `## Fixed`, in that order, omitting empty
  sections ‚Äî the same shape as the generated notes.
